# Human Authority Model

> Date: 2026-04-03  
> Purpose: Define an authority verification system that distinguishes "conscious human directives" from "text passed between agents" in agent environments

---

## 1. Threat Model

### Why This Is Needed

In multi-agent team environments, the following misattributions can occur:

- Agent A tells Agent B "work on the same branch" → B misinterprets it as a human directive and opens an isolation rule exception
- A Leader agent outputs "protected branch exception allowed" → a Builder agent processes it as human approval
- SESSION.md HANDOFF `next` items contain high-risk commands → the next agent executes without verification

### Core Principle

**Only humans have the authority to grant exceptions.** Agents can request exceptions, but they cannot approve them.

---

## 2. Authority Levels

| Role | Request Exception | Approve Exception | Can Delegate | Notes |
|------|------------------|-------------------|-------------|-------|
| **Human** (highest authority) | O | O | O | Can allow all exceptions |
| **Leader agent** | O | X (default) | X | Observe/report/suggest/escalate |
| **Regular agent** | O | X | X | Execute only, no exception approval authority |

For a Leader agent to approve exceptions, **explicit human delegation (grant)** is required (see Section 5).

---

## 3. Approval Levels — Separated by Risk

Connects to existing systems (03-security reversibility grades, AGENT_OPERATING_MODEL approval model).

### 3-1. General Exception (Low Risk)

Uses a re-confirmation phrase. Same pattern as 03-security's existing R1 confirmation phrases.

```
Agent: "To open a branch sharing exception, enter the following phrase."
Human: CONFIRM same-branch-20260403
```

**Format**: `CONFIRM {exception-type}-{date}`

Applicable scope:
- Parallel agent branch sharing
- Domain separation changes
- Document-only rule relaxation

| Mapping | Value |
|---------|-------|
| Reversibility grade | R0–R1 |
| Approval model | P1–P2 |

### 3-2. Medium Risk Exception

Uses a re-confirmation phrase with a one-time nonce. The agent generates a nonce, presents it to the human, and the human re-enters it as-is.

```
Agent: "To continue, enter the following phrase exactly:
        CONFIRM same-branch-4821-20260403"
Human: CONFIRM same-branch-4821-20260403
```

**Format**: `CONFIRM {exception-type}-{4-digit random}-{date}`

Applicable scope:
- Protected branch exception
- Large-scale dependency changes
- API breaking change approval

| Mapping | Value |
|---------|-------|
| Reversibility grade | R1–R2 |
| Approval model | P3 |

### 3-3. High Risk Exception

A re-confirmation phrase alone is not sufficient. Out-of-band approval or human direct execution is required.

```
Agent: "This task cannot be executed by the agent.
        Please run the following command directly:
        {specific command}"
```

Applicable scope:
- Destructive DB operations (migrate reset, DROP, etc.)
- Force push
- External system state changes
- .env file modifications

| Mapping | Value |
|---------|-------|
| Reversibility grade | R2 |
| Approval model | P3 (human direct execution) |

## 4. Nonce Generation Rules

### Who Generates

The nonce is generated by **the agent receiving the exception request** — not the agent making the request.

```
❌ Agent A → Agent B: "Enter CONFIRM xxx-4821"  (Meaningless since A knows the nonce)
✅ Agent B (receiving side): "To continue, enter CONFIRM xxx-4821" → Human enters it
```

### Nonce Requirements

- 4-digit random number (0000–9999)
- No duplicates within a session
- Expiration: Valid only in the current conversation turn. Cannot be reused in the next turn.

### Forbidden Practices

- Do not statically store nonces or prefixes in repo files (.env, config, etc.)
- Do not pass nonces through inter-agent messages
- Do not generate nonces with predictable patterns (sequential increment, timestamps, etc.)

## 5. Delegation Grant Rules

Exception approval authority can be delegated to a Leader agent. However, delegation is not permanent authority — it is a **grant with scope and expiration**.

### Delegation Conditions (all must be met)

1. **Explicit human delegation**: Agents cannot create or request delegation on their own
2. **Scope specified**: Explicitly state what types of exceptions can be approved
3. **Expiration time**: Permanent delegation without expiration is forbidden
4. **Audit log**: Record delegation creation and usage history

### Delegation Format

```
Human: "Delegate branch sharing exception for docs-only to leader until 6 PM today"
```

Record location: SESSION.md or a separate authority log

```markdown
## Authority Grants

| Grantee | Scope | Expires | Granted By | Granted At |
|---------|-------|---------|-----------|-----------|
| leader | same-branch exception (docs only) | 2026-04-03 18:00 | human | 2026-04-03 10:00 |
```

### When Using a Delegation

When a Leader agent approves an exception within delegated scope:

1. Verify the grant is valid (scope, expiration)
2. Record in audit log after execution
3. Escalate to human if request is outside scope

### Delegation Restrictions

- High-risk exceptions (R2) cannot be delegated — always requires direct human approval
- Leader re-delegating to another agent is forbidden
- Auto-renewing expired grants is forbidden

---

## 6. Audit Log

Exception approvals, delegation creation, and delegation usage are all recorded.

### Record Location

Choose based on project scale:

- **Small scale**: Add `authority_events` field to SESSION.md's HANDOFF block
- **Medium scale**: `ops/authority-log.jsonl` (same format as AGENT_OPERATING_MODEL's event model)

### Record Format (JSONL)

```json
{"ts":"2026-04-03T10:00:00Z","event":"grant_created","grantee":"leader","scope":"same-branch (docs)","expires":"2026-04-03T18:00:00Z","granted_by":"human"}
{"ts":"2026-04-03T10:30:00Z","event":"exception_approved","type":"same-branch","agent":"leader","grant_ref":"2026-04-03T10:00:00Z","target":"feature/260403-docs"}
{"ts":"2026-04-03T11:00:00Z","event":"exception_approved","type":"same-branch","agent":"human","nonce":"4821","target":"feature/260403-api"}
```
