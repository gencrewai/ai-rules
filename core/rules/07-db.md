# 07-db — Database Safety Rules

## Priority

This rule has the **second-highest priority after 03-security**. DB data loss is irreversible.

---

## 1. DB Name Collision Prevention (CRITICAL)

When multiple projects share the same local PostgreSQL instance,
**a name collision means one project's `migrate reset` can completely destroy another project's data.**

### Real Incident (2026-04-01)

`my-studio-app` (old repo) and `my-studio-app-v2` (current repo) both used `localhost:5432/myapp_dev`.
A migration operation in the old repo wiped the DB, destroying all user data in the current repo.

### DB Naming Rules

- **Every project must have a unique DB name**
- Format: `{project_name_purpose}` (e.g., `myapp_dev`, `my_saas_app_v2`, `my_store_local`)
- Forbidden: `postgres`, `test`, `dev`, `app`, `database`, `mydb`
- Even if two repos serve the same product — **separate repo = separate DB**

### Collision Check on New Project Setup (agent must run)

```bash
# 1. List current local databases
psql -U postgres -c "\l" | grep -v template

# 2. Check DB names used by other projects
find ~/dev -name ".env" -not -path "*/node_modules/*" 2>/dev/null \
  | xargs grep "DATABASE_URL" 2>/dev/null
```

If a collision is found, **report to user immediately** — suggest an alternative name — proceed only after user approval.

### Prisma Safety Environment Variable (required for new project setup)

Prisma v6.15.0+ automatically blocks destructive commands (`migrate reset`, `db push --force-reset`)
in agent environments when `PRISMA_USER_CONSENT_FOR_DANGEROUS_AI_ACTION` is not set.

Must be included in new project `.env.example`:
```bash
# Prisma AI safety guard — destructive agent commands are blocked when this is unset
# Set to "I understand the risks" only during initial local setup (by a human)
PRISMA_USER_CONSENT_FOR_DANGEROUS_AI_ACTION=
```

## 2. Destructive Migration Commands Forbidden (CRITICAL)

**Governance hook (`guard-branch.sh`) + tooling hooks automatically block `migrate reset`, `db push --force-reset`, and `DROP DATABASE/TABLE` patterns.**
The table below is maintained as an agent decision reference:

### Agent Must Never Execute

| Command | Severity | Reason |
|---------|----------|--------|
| `prisma migrate reset` | 🔴 Critical | Drops and recreates entire DB — all data lost |
| `prisma migrate dev` | 🟠 High | May cause data loss when auto-resolving drift via shadow DB |
| `prisma db push --force-reset` | 🔴 Critical | Force-resets schema |
| `alembic downgrade` | 🟠 High | Migration rollback — may drop columns/tables |
| `alembic downgrade base` | 🔴 Critical | Full migration rollback — complete schema reset |
| `DROP DATABASE` | 🔴 Critical | Deletes entire database |
| `DROP TABLE` | 🔴 Critical | Deletes table |
| `TRUNCATE` | 🟠 High | Deletes all data in table |
| `DELETE FROM` (no WHERE) | 🔴 Critical | Deletes all rows |

### Agent Allowed Commands

| Command | Description |
|---------|-------------|
| `prisma migrate status` | Status check only (read-only) |
| `prisma migrate deploy` | Applies only pending migrations in order (preserves data) |
| `prisma generate` | Regenerates client code only |
| `alembic upgrade head` | Applies pending migrations in order |
| `alembic current` | Checks current revision only |

### `prisma migrate dev` Usage Criteria (Exception)

`prisma migrate dev` is forbidden by default since it auto-resolves drift via shadow DB.
However, it may be run **by a human** (never by agent) when **all** conditions below are met:

**New project (no data to preserve in DB)**
```
✅ Allowed when:
- Local-only DB with no production/staging deployment history
- migrate status detects drift or missing migrations

Agent guidance format:
"This is a new project with no data to preserve.
 Please run the following command manually:
   cd {path} && npx prisma migrate dev --name {desc}
 Commit the generated migration file afterward."
```

**Existing project (has production/staging deployment history) — Forbidden**
```
❌ Reason: Data loss possible when shadow DB auto-resolves drift

Schema change procedure for existing projects:
1. prisma migrate status  — check current state and report to user
2. Modify schema.prisma
3. Agent drafts migration SQL — human reviews
4. prisma migrate deploy  — apply (preserves data)
```

---

### When User Requests a Forbidden Command

```
When user requests "run prisma migrate reset":

❌ Never execute directly

✅ Show warning and wait for confirmation phrase (03-security reversibility R2):
"⚠️ migrate reset will delete ALL data in the database. (R2 — irreversible)
Current DB: {DB_NAME} @ {HOST}

Backup strongly recommended before proceeding:
  pg_dump -U postgres {DB_NAME} > backup_$(date +%Y%m%d_%H%M).sql

To continue, type the following phrase exactly:
> CONFIRM reset-{DB_NAME}-{today's date}
Example: CONFIRM reset-myapp-dev-20260401"
```

## 3. Migration Execution Process (Agent Standard)

### Required Order After Schema Changes

```
1. prisma migrate status  — check current state and report to user
2. Modify schema.prisma
3. Provide migration file creation instructions (human executes)
4. After user approval: prisma migrate deploy
5. prisma generate
```

### Pre-Migration Report Format

```
### Migration Execution Plan
- Target DB: myapp_dev @ localhost:5432
- Changes: Adding layoutSchema Json? column to Page model
- Migration file: 20260401_add_layout_schema_to_page
- Data impact: None (adding optional column)

Command:
  cd backend/api && npx prisma migrate deploy

Proceed?
```

For changes with potential data loss (column removal, type change, NOT NULL addition):
```
⚠️ Potential data loss
- Column being dropped: old_column (existing data permanently deleted)
- Backup recommended before proceeding
```

---

## 4. Migration File Protection

- Migration files must **never be modified or deleted** — add changes via new migrations
- Never regenerate migrations locally with `migrate dev` after production deployment
- **Schema change = always a new migration file** — never just modify `schema.prisma` and stop
- Migration file naming: `{timestamp}_{description}` (e.g., `20260401011547_add_layout_schema_to_page`)

---

## 5. Migration Commands by Environment

```bash
### Local development
cd backend/api
npx prisma migrate deploy   # Prisma
alembic upgrade head        # Python/Alembic

### Staging/Production (Docker)
docker compose exec api npx prisma migrate deploy
docker compose exec backend alembic upgrade head

### Railway / Remote
railway run npx prisma migrate deploy
```

## 6. Local Backup Rules

### Backup Before Destructive Operations (agent instructs, human executes)

```bash
# PostgreSQL backup
pg_dump -U postgres {DB_NAME} > backup_$(date +%Y%m%d_%H%M).sql

# Restore
psql -U postgres {DB_NAME} < backup_20260401_1200.sql
```

### Automated Backup Script

Recommended: add `scripts/backup-db.sh` to the project (see `TOOLING_SETUP.md`).
Backup files should be in `.gitignore` (`*.sql`, `backups/`).

```markdown

- Migration file: 20260401011547_add_layout_schema_to_page

### Commands by Environment

# Production: docker compose exec api npx prisma migrate deploy

### Data Impact
- [No impact] Optional column added, existing rows remain null
```

If there is potential data loss:
```
### ⚠️ Potential Data Loss
- Reason: old_column column being dropped
- Backup recommended before proceeding
- Awaiting human approval
```
