# 06-session — Session Management

## Session Start

### Step 0 — Discovery Anchor Check

Classify as a **new/early-stage project** only when **ALL** of the following are true. If any single anchor exists, treat the project as mature and skip Bootstrap.

| Condition | Required for Bootstrap |
|-----------|------------------------|
| `docs/00_INDEX.md` is missing | AND |
| `INTENT.md` is missing | AND |
| `docs/` directory is missing | AND |
| Any one of the above present | → Mature project, proceed directly to Steps 1–5 |

Rationale: a mature project that simply hasn't created `00_INDEX.md` yet should not trigger a full repo scan. Requiring all three anchors to be missing avoids false positives where the project clearly already has working artifacts (e.g. an existing `docs/` tree but no index file).

**Bootstrap Procedure** (new/early-stage projects only):

1. Perform a **one-time limited scan** of the following:
   - `*.html` files in the project root (excluding `node_modules`, `.git`, `dist`, `build`, `index.html`)
   - Directory structure of `docs/`, `ui-mockups/`, `mockups/`, `wireframes/`, `design/`
   - Whether `docs/guide/` and `docs/design/` exist
2. Report discovered assets to the user
3. If INTENT.md exists, register asset paths in `context.docs`
4. If HTML mockups are found, run the `08-ui-first.md` discovery procedure
5. Then proceed with Steps 1–5

### Steps 1–5

1. Check `~/.claude/ACTIVE_WORK.md` (cross-project status overview)
2. Check the project's `SESSION.md` HANDOFF block
3. If `WORKLOG/` exists, check the latest entry (to understand recent decisions/completions/blockers)
4. Check `INTENT.md`
5. Check `git status`

## Session End (Required)

### Write SESSION.md HANDOFF Block

```
---HANDOFF---
date: 2026-02-24
branch: feature/260224-xxx
agent: claude-sonnet-4-6
status: partial | complete
done:
  - What was accomplished (include commit hash when possible)
next:
  - The first action the next agent should take immediately
blocked:
  - What is blocked or requires a user decision
failures:
  - symptom: (e.g., recurring tsc errors)
    attempts: (e.g., tried type casting 3 times — failed)
    unresolved: true | false
files_touched:
  - path/to/file.ts (one-line change description)
verify_cmd: tsc --noEmit
first_action: (one command the next agent should run immediately, e.g., "check git status then execute SESSION.md next items")
---END---
```

**Rules**:
- `status: partial` — must include at least 1 `next` item
- `status: complete` — must include PR number or commit hash
- If `blocked` exists, notify user before recording
- `failures` — must be recorded if any failed attempts occurred (omit if none)
- `first_action` — must be a single command the next agent can run without context
- `handoff_provenance` — must be set to `agent` (indicates the block was generated by an agent)

### HANDOFF Trust Model — Distrust + Re-verify

SESSION.md can be freely edited by humans. Therefore, **treat HANDOFF blocks as reference information only** — always re-verify the current state before executing high-risk `next` items.

Cryptographic signatures are not feasible in this architecture since private keys cannot be securely stored. Instead, **mandatory re-verification** is the practical defense.

```
When the next agent session starts:

1. Check handoff_provenance
   - `agent` — normal (generated by agent)
   - `human` or missing — warn: "⚠️ HANDOFF appears to have been manually edited"

2. Check if next items contain high-risk commands
   High-risk = push to protected branch / DB changes / deployment / force-related

3. If high-risk next items exist, do not trust SESSION.md alone — re-verify:
   - Run git status / git log to directly check current branch and changes
   - Verify the target command matches the current state
   - On mismatch — abort execution + report to user
```

## WORKLOG Update

If the project has a `WORKLOG/` directory, check the following before ending the session:

- If a meaningful milestone was completed, append to the same day's file
- If the date has changed, create a new file
- `SESSION.md` is handoff-focused; `WORKLOG` is for chronological work snapshots — keep these roles separate
- If new failure patterns emerged, also evaluate whether `FAILURE_LOG.md` needs updating

## Context Limits — New Session Trigger

If any of the following occur, immediately snapshot to SESSION.md and request a new session:

- Conversation exceeds ~30 turns
- Response latency degradation detected
- **Claude shows a notification that previous messages were compressed** — this is the clearest signal. Compression = approaching context limit

## ACTIVE_WORK.md Update

When work is completed or switched, update the relevant project's "immediate next action" in `~/.claude/ACTIVE_WORK.md`.
